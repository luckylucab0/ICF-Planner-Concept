# Projekt-Konventionen für KI-Sessions

Diese Regeln gelten verbindlich für jede Session, die in diesem Repository arbeitet.

## Attribution (wichtigste Regel)

- **Autor/Committer aller Commits:** `luckylucab0 <luckylucab0@users.noreply.github.com>`.
  Zu Session-Beginn setzen: `git config user.name luckylucab0 && git config user.email luckylucab0@users.noreply.github.com`
- **Keine KI-Verweise in Repo-Artefakten** – nirgends: keine `Co-Authored-By:`-Zeilen,
  keine „Generated by/with …"-Footer, keine Session-Links, keine Tool-Namen in
  Commit-Messages, PR-Titeln/-Beschreibungen, Issue-Kommentaren, Code-Kommentaren
  oder Doku.
- Der GitHub-Anbindungsdienst hängt an PR-Beschreibungen automatisch einen
  „Generated by …"-Footer an. **Direkt nach jedem PR-Erstellen** die Beschreibung
  per `update_pull_request` ohne diesen Footer neu setzen und per Read-back prüfen.

## Branch-Modell

- `main` = Produktionscode. Geschützt: nur per PR mit grünen Pflicht-Checks
  (`checks`, `build`, `e2e`, Trivy, CodeQL) änderbar, keine Direkt-Pushes.
- `dev` = Arbeitsbranch. Alle Entwicklung passiert hier; Änderungen gehen per
  PR `dev → main` in Produktion.
- Keine zusätzlichen Branches anlegen, außer der Nutzer verlangt es. Branch-Namen
  beschreiben den Inhalt (z. B. `feature/…`, `fix/…`) – niemals Tool-Namen.

## Commits & Releases

- **Conventional Commits** sind Pflicht (`feat:`, `fix:`, `docs:`, `chore:` …) –
  release-please generiert daraus Changelog und Versionen.
- CI muss grün sein; Qualitätsregeln siehe [CONTRIBUTING.md](../CONTRIBUTING.md)
  (i18n statt hartcodierter Strings, serverseitige Berechtigungen mit Negativtests,
  keine Secrets im Code).
